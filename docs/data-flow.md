# Backend Data Flow

This document traces a single request through Anki's layers at _runtime_ and
describes the machinery that carries it: the numeric dispatch contract, the
Python↔Rust FFI boundary, the raw SQL proxy, error conversion, progress
reporting, and the `OpChanges`/undo model that drives UI refresh.

It complements two neighbouring docs that you should read first:

- [architecture.md](./architecture.md) — the high-level layer split.
- [language_bridge.md](./language_bridge.md) — how to _declare, call, and
  implement_ a new RPC. This document instead explains what happens to a call
  once it is made.

See [protobuf.md](./protobuf.md) for how the message and service definitions in
`proto/anki/` are written.

## Layer overview

```
Svelte / TypeScript  (ts/)            ─┐
       │  postProto() over HTTP POST   │  GUI
PyQt / Python GUI    (qt/aqt/)        ─┘
       │  Collection / aqt helpers
Python library       (pylib/anki/)    ─┐
       │  RustBackend._run_command()   │  library
rsbridge (PyO3 FFI)  (pylib/rsbridge/)─┘
       │  Backend::run_service_method()
Rust core            (rslib/)          ── service impl → storage
       │
SQLite collection file (.anki2)
```

Calls only ever travel **downward**. The Rust core never calls up into Python or
TypeScript (it may call out to AnkiWeb and other servers, which is out of scope
here). Python calling into TypeScript is possible but discouraged — see the
"Invoking a TypeScript method from Python" section of
[language_bridge.md](./language_bridge.md).

## The numeric dispatch contract

Every backend RPC is addressed by a `(service_index, method_index)` pair of
integers rather than by name. Both indices are derived from the protobuf
descriptor pool at build time and are guaranteed to be **identical across every
language**, so a client only needs to send `(service, method, request_bytes)`
and read back `response_bytes`.

The indices are produced by
[`get_services()`](../rslib/proto_gen/src/lib.rs) in `rslib/proto_gen`, which
parses the `.proto` files and returns two lists:

- **`CollectionService`** — services whose methods run with a `Collection`
  open (the common case, e.g. `DecksService`, `NotesService`).
- **`BackendService`** — services prefixed `Backend…` whose methods run on the
  `Backend` itself, with no collection required (e.g. opening a collection,
  i18n, sync setup).

For a `BackendFooService`, any method that is _not_ explicitly overridden is
copied in as a **delegating method** (`delegating_methods` in
[proto_gen/src/lib.rs](../rslib/proto_gen/src/lib.rs)) that forwards to the
corresponding `FooService` method. This is why most services only define the
collection-scoped variant.

These indexed services then drive code generation in three places:

- Rust traits + dispatcher — [rslib/rust_interface.rs](../rslib/rust_interface.rs)
- Python methods — [rslib/proto/python.rs](../rslib/proto/python.rs)
- TypeScript wrappers — the `@generated/backend` module

## Python → Rust path

Take `Collection.add_note(note, deck_id)` as a worked example. Its real
signature lives in
[pylib/anki/collection.py](../pylib/anki/collection.py) and returns an
`OpChangesWithCount`. The call descends as follows:

1. **High-level API** — `Collection.add_note()`
   ([pylib/anki/collection.py](../pylib/anki/collection.py)) is the Pythonic
   wrapper that add-ons and the GUI should call. Per
   [language_bridge.md](./language_bridge.md), backend methods are always
   reached through such a helper rather than touching `_backend` directly.

2. **Generated stub** — it calls the generated
   `RustBackendGenerated.add_note()` in `out/pylib/anki/_backend_generated.py`.
   That file is produced by [rslib/proto/python.rs](../rslib/proto/python.rs),
   which emits a typed `foo(**kwargs)` method plus a raw `foo_raw(bytes)`
   method for each RPC. Both end in a call like
   `self._run_command(service_idx, method_idx, message.SerializeToString())`.

3. **Command entry point** — `RustBackend._run_command(service, method, input)`
   ([pylib/anki/_backend.py](../pylib/anki/_backend.py)) is the single Python
   chokepoint. It calls `self._backend.command(service, method, input)` and
   converts any returned error into a Python exception (see _Errors_ below).

4. **FFI boundary** — `_rsbridge.Backend.command()`
   ([pylib/rsbridge/lib.rs](../pylib/rsbridge/lib.rs)) is the only FFI surface
   between Python and Rust. It releases the GIL (`py.detach`) and calls
   `self.backend.run_service_method(service, method, in_bytes)`.

5. **Rust dispatcher** — `Backend::run_service_method(service, method, input)`
   is generated by [rslib/rust_interface.rs](../rslib/rust_interface.rs) and
   included into the crate via
   [rslib/src/services.rs](../rslib/src/services.rs)
   (`include!(concat!(env!("OUT_DIR"), "/backend.rs"))`). It matches on the two
   indices and, for collection-scoped services, acquires the collection lock via
   `Backend::with_col()` ([rslib/src/backend/mod.rs](../rslib/src/backend/mod.rs)).

6. **Service implementation** — the dispatcher decodes the protobuf request and
   calls the trait method implemented in `rslib/src/<module>/service.rs` (for
   notes, [rslib/src/notes/service.rs](../rslib/src/notes/service.rs)). The
   implementation does the real work against the storage layer and returns a
   protobuf response, which is re-encoded to bytes and handed back up the same
   chain.

## TypeScript → Rust path

The web frontend reaches the _same_ backend over HTTP rather than through the
FFI bridge:

1. Generated wrappers in `@generated/backend` call
   [`postProto()`](../ts/lib/generated/post.ts), which serializes the protobuf
   request and `POST`s it to `/_anki/{method}`.

2. The request hits the Flask media server,
   [qt/aqt/mediasrv.py](../qt/aqt/mediasrv.py). Its catch-all route
   `handle_request` (`@app.route("/<path:pathin>", methods=["GET", "POST"])`)
   dispatches `POST /_anki/{method}` into the backend
   (`_extract_collection_post_request`), running the work off the main thread.

3. From there the call enters the backend exactly like the Python path above.

In development, the SvelteKit pages are served by Vite on port `5173` and the
Flask media server on port `40000` proxies the protobuf endpoints; in
production the pages are pre-built into `out/sveltekit/` and served directly.
See [e2e-testing.md](./e2e-testing.md) and [architecture.md](./architecture.md)
for the serving details rather than relying on this summary.

## The Speedrun landing screen (`"manifold"` state)

The Speedrun fork changes which screen the main window opens on. `AnkiQt`
([qt/aqt/main.py](../qt/aqt/main.py)) has a `MainWindowState` string-literal
enum; the fork adds a `"manifold"` member with `setupManifold()` /
`_manifoldState()`, and `loadCollection` now calls `moveToState("manifold")`
instead of landing on the deck list. The state is served by the
`Manifold` webview ([qt/aqt/manifold.py](../qt/aqt/manifold.py)), a home screen
modeled on [qt/aqt/deckbrowser.py](../qt/aqt/deckbrowser.py) whose HTML is built
by `build_manifold_html(col, depth)` in [qt/aqt/pgre.py](../qt/aqt/pgre.py). The
manifold is drawn from its actual equations rather than an image: `pgre` meshes
the Fermat-quintic Calabi-Yau cross-section (`z1^n + z2^n = 1`, Hanson's
parametrization, `n = 5`), projects and Lambert-shades it, and emits inline SVG
polygons (`_manifold_geometry`, computed once and cached since it's
collection-independent). The 10 button positions are the spike tips _detected
from the projected geometry itself_ (`_detect_tips`), so a button always lands on
a spike. Decks are shown 9-per-page and paged
through by "depth": depth `n` shows decks `9n..9n+8` on the deck spikes and
leaves any leftover spikes empty; the 10th spike is always a "More decks" button
(`pycmd('more')`) that advances to the next depth, wrapping back to depth 0 past
the last page. `Manifold` ([qt/aqt/manifold.py](../qt/aqt/manifold.py)) holds the
current `depth` (reset to 0 on show).

Clicks flow through the standard `pycmd()` WebView bridge:

- `open:<deckId>` — selects the deck and moves to its "Study Now" overview
  (same effect as clicking a deck in the classic list).
- `more` — advances to the next depth (page of 9 decks).
- `classic` — opens the intact classic deck browser (`DeckBrowser`).

The toolbar "Decks" link ([qt/aqt/toolbar.py](../qt/aqt/toolbar.py)
`_deckLinkHandler`), the `d` shortcut, the overview "Decks" back-link
([qt/aqt/overview.py](../qt/aqt/overview.py)), the reviewer "Finish" action
([qt/aqt/reviewer.py](../qt/aqt/reviewer.py)), and the overview no-deck fallback
all return to `"manifold"`.

On first launch of a fresh collection, `loadCollection` also calls
`_seed_default_decks`, which runs `maybe_import_default_decks` /
`import_default_decks` in [qt/aqt/pgre.py](../qt/aqt/pgre.py) to auto-import the
9 bundled PGRE `.apkg` decks (guarded by the collection config flag
`pgreDefaultDecksImported`; idempotent and one-time per collection). The decks
ship in `qt/aqt/data/decks/`, copied into the build output by the `build_decks`
rule in [build/configure/src/aqt.rs](../build/configure/src/aqt.rs).

## The raw SQL proxy (`db_command`)

Some callers need to run SQL directly instead of going through a typed RPC
(interactive tools, a few internal utilities, and AnkiDroid). For these,
`RustBackend.db_query()` / `db_command()`
([pylib/anki/_backend.py](../pylib/anki/_backend.py)) bypass protobuf entirely
and send a **JSON-encoded** request to
`Backend::run_db_command_bytes()`
([rslib/src/backend/mod.rs](../rslib/src/backend/mod.rs)), implemented in
[rslib/src/backend/dbproxy.rs](../rslib/src/backend/dbproxy.rs). The proxy
deserializes a `DbRequest`, runs it against the collection's SQLite connection,
and serializes a `DbResult` back with `serde_json`.

JSON is used here (rather than protobuf) because the row shapes are dynamic and
this path is performance-sensitive for bulk queries.

## Errors

Errors flow back up as data, then become exceptions at the Python boundary:

1. Rust code returns `AnkiError`/`Result` (see
   [rslib/src/error/mod.rs](../rslib/src/error/mod.rs); `AnkiError::message()`
   localizes the message via `I18n`).

2. When a service method fails, the backend converts the error with
   `AnkiError::into_protobuf(tr)`
   ([rslib/src/backend/error.rs](../rslib/src/backend/error.rs)), mapping each
   variant to a `BackendError` `Kind` and a localized description.

3. The encoded `BackendError` is returned as the error half of the
   `Result<Vec<u8>, Vec<u8>>` from `run_service_method`.

4. `RustBackend._run_command()` decodes it and raises the matching Python
   exception from the hierarchy in [pylib/anki/errors.py](../pylib/anki/errors.py)
   (`AnkiException` → `BackendError` → `NotFoundError`, `DBError`, `SyncError`,
   `TemplateError`, `InvalidInput`, `SearchError`, …).

## Progress reporting

Long-running operations (sync, import/export, database check) cannot stream
progress through the synchronous command channel, so they publish it to shared
state instead. A `ThrottlingProgressHandler` writes into the
`ProgressState` held by the backend
([rslib/src/progress.rs](../rslib/src/progress.rs)), and the UI polls it through
the `latest_progress` RPC
([rslib/src/collection/service.rs](../rslib/src/collection/service.rs),
[rslib/src/backend/collection.rs](../rslib/src/backend/collection.rs)).

## Operations, `OpChanges`, and undo

Anki's mutating operations don't just return their result — they return a
description of _what changed_, so the GUI knows which views to refresh and the
undo system knows what to record.

- Operations return an `OpOutput<T>` carrying both the result and an
  `OpChanges`. `OpChanges` pairs an `Op` (an enum of ~every user-visible
  operation: `AddNote`, `AnswerCard`, `UpdateDeck`, …) with a `StateChanges`
  struct of boolean flags (`card`, `note`, `deck`, `tag`, `notetype`, `config`,
  `deck_config`, `mtime`). All three live in
  [rslib/src/ops.rs](../rslib/src/ops.rs).

- The Python side surfaces these as `OpChanges*` return types (e.g. `add_note`
  returns `OpChangesWithCount`), and the GUI's operations layer in
  [qt/aqt/operations/](../qt/aqt/operations/) uses the flags to decide which UI
  components to redraw.

- The same operations are recorded by the `UndoManager`
  ([rslib/src/undo/](../rslib/src/undo/)), which keeps undo/redo stacks as a
  transaction log so any tracked change can be reversed.

This "every change announces itself" pattern is the reason most backend
mutations are wrapped in a transaction and return an `OpChanges` rather than a
bare value.
